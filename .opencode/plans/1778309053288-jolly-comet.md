# Plan: Drizzle + Neon WhatsApp Chat Archive

## Decisions

- Use Drizzle with Neon Postgres via `DATABASE_URL`.
- Use Vercel Blob for decrypted media storage.
- Model participants from WhatsApp sender/chat identifiers only; no app auth yet.
- A participant can have one current reaction per target message. New reaction webhook events update the existing row.
- Build a local shadcn-style chat UI, not an external ChatCN registry install.
- Replace filesystem persistence in `src/lib/whatsapp-webhook.ts`; do not delete existing `whatsapp/` files automatically.

## Schema

Create `src/db/schema.ts` with these tables.

### `participants`

Stores normalized WhatsApp identities.

- `id uuid primary key default gen_random_uuid()`
- `jid text not null unique`
- `display_name text`
- `phone_number text`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`
- Index: unique `jid`

### `messages`

Stores actual chat messages. Reaction events do not become chat bubbles; they update `message_reactions`.

- `id uuid primary key default gen_random_uuid()`
- `whatsapp_message_id text not null unique`
- `chat_jid text not null`
- `sender_participant_id uuid not null references participants(id)`
- `message_type text not null`
- `body text`
- `caption text`
- `sent_at timestamptz`
- `received_at timestamptz not null default now()`
- `reply_to_message_id uuid references messages(id)`
- `reply_to_whatsapp_message_id text`
- `reply_to_participant_jid text`
- `quoted_text text`
- `raw_payload jsonb not null`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`
- Index: unique `whatsapp_message_id`
- Index: `(chat_jid, sent_at)`
- Index: `sender_participant_id`
- Index: `reply_to_message_id`
- Index: `reply_to_whatsapp_message_id`

Keep both reply IDs because a reply can arrive before the parent message.

### `message_media`

Stores Blob-backed media for images, videos, audio, documents, stickers, and albums.

- `id uuid primary key default gen_random_uuid()`
- `message_id uuid not null references messages(id) on delete cascade`
- `media_type text not null`
- `mime_type text`
- `file_name text`
- `file_size integer`
- `sha256 text`
- `blob_url text not null`
- `blob_path text not null`
- `width integer`
- `height integer`
- `duration_seconds integer`
- `sort_order integer not null default 0`
- `raw_media_payload jsonb`
- `created_at timestamptz not null default now()`
- Index: `message_id`
- Index: `media_type`
- Unique: `(message_id, sort_order)`

Do not deduplicate by `sha256` initially unless the webhook payload proves it is reliable across all media types.

### `message_reactions`

Stores current reaction state per participant and target message.

- `id uuid primary key default gen_random_uuid()`
- `message_id uuid references messages(id) on delete cascade`
- `target_whatsapp_message_id text not null`
- `participant_id uuid not null references participants(id)`
- `emoji text`
- `reacted_at timestamptz`
- `raw_payload jsonb not null`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`
- Unique: `(target_whatsapp_message_id, participant_id)`
- Index: `message_id`
- Index: `participant_id`

Webhook handling uses an upsert on `(target_whatsapp_message_id, participant_id)`. If WhatsApp sends a removal as an empty/null emoji, update the row to `emoji = null`.

### `webhook_events`

Stores raw webhook attempts for idempotency and debugging.

- `id uuid primary key default gen_random_uuid()`
- `event_id text`
- `event_hash text not null unique`
- `source text not null default 'whatsapp'`
- `status text not null default 'received'`
- `processed_at timestamptz`
- `error text`
- `raw_payload jsonb not null`
- `created_at timestamptz not null default now()`
- `updated_at timestamptz not null default now()`
- Unique: `event_hash`
- Index: `(status, created_at)`

Use `event_hash` from the raw request body because the current payload shape does not expose a clearly stable event ID.

## Webhook Flow

1. In `src/app/api/webhooks/whatsapp/route.ts`, keep `x-webhook-signature === WHATSAPP_WEBHOOK_SECRET` validation.
2. Read the request body as text, parse JSON from that text, and pass both to `processWhatsAppWebhook`.
3. In `processWhatsAppWebhook`, compute a deterministic hash of the raw body.
4. Upsert or insert a `webhook_events` row.
5. If an existing row is already `processed`, return early.
6. If the payload has no message, store the event as processed with no message rows.
7. If the message is a reaction, upsert `message_reactions` and resolve `message_id` if the target message already exists.
8. If the message is a chat message, upsert participant, upsert message by `whatsapp_message_id`, upload media only when media rows are missing, and resolve reply links.
9. Mark the webhook event as `processed` on success.
10. On error, mark it `failed` with the error message and rethrow so the sender can retry.

Message and reaction unique constraints provide message-level idempotency even if raw webhook hashes differ between retries.

## Participant Derivation

Use the sender JID as the stable participant identity.

Display-name priority:

- `pushName`
- participant phone/JID-derived value
- sender JID fallback

Phone number:

- derive from `123456789@s.whatsapp.net` when possible
- leave null for groups, broadcasts, and unknown JID formats

Upsert participants by `jid`. Update `display_name` only when the incoming value is non-empty and better than a fallback.

## Media And Vercel Blob

Move current media download/decrypt logic out of `src/lib/whatsapp-webhook.ts` into `src/lib/whatsapp-media.ts`.

Flow:

1. Extract all media entries from the message, not just the first one.
2. Download using `url` or `directPath` as today.
3. Decrypt with the existing HKDF/AES logic.
4. Build a stable Blob path like `whatsapp/{safeChatJid}/{safeMessageId}/{sortOrder}-{safeFileName}`.
5. Upload with Vercel Blob using the detected MIME type.
6. Insert `message_media` rows with `blob_url`, `blob_path`, metadata, and raw media payload.
7. On duplicate webhook delivery, skip upload when media rows already exist for the message.

If Blob upload fails, fail the webhook event so a retry can complete media persistence. Do not write decrypted media to local disk.

## Reply Resolution

When inserting a reply:

- Store `reply_to_whatsapp_message_id`, `reply_to_participant_jid`, and `quoted_text` from `contextInfo`.
- If the parent message exists, set `reply_to_message_id` immediately.
- After inserting any message, update unresolved messages where `reply_to_whatsapp_message_id` matches the inserted `whatsapp_message_id`.

## Chat Queries

Create `src/lib/chat-queries.ts` with server-only query helpers:

- `getChats()`: distinct chats with latest message preview, latest timestamp, and participant count if cheap.
- `getChatMessages(chatJid?: string)`: messages with sender participant, media, reactions, and reply target summary.

Ordering:

- `sent_at asc nulls last`
- `received_at asc`
- `created_at asc`

Use Server Components for DB reads. Local Next docs confirm Server Components can query an ORM/database without leaking credentials to the client.

## Verification

Run:

- `bun install`
- `bun run db:generate`
- `bun run db:migrate`
- `bun run lint`
- `bun run build`

Manual webhook checks:

- Send the same text message webhook twice; expect one `messages` row.
- Send the same media webhook twice; expect one message and one media set, with no duplicate upload when media rows already exist.
- Send reaction A from participant X to message Y; expect one `message_reactions` row.
- Send reaction B from the same participant X to message Y; expect that same row updated.
- Send a reaction removal if supported by the provider; expect the same row with `emoji = null`.
- Send a reply where the parent exists; expect `reply_to_message_id` resolved.
- Send a reply before the parent; expect it unresolved first and resolved after the parent arrives.
- Send image, video, audio, document, and sticker messages; expect Blob URLs stored and UI rendering.
- Load `/`; expect chronological chat bubbles with participants, media, replies, and reactions.

Production checks:

- Confirm `DATABASE_URL`, `BLOB_READ_WRITE_TOKEN`, and `WHATSAPP_WEBHOOK_SECRET` exist in Vercel env.
- Confirm invalid webhook signatures return `401`.
- Confirm large media does not exceed Vercel function time or memory limits.

## Risks

- WhatsApp reaction removal semantics need real payload confirmation.
- Large videos may exceed serverless limits during download/decrypt/upload.
- Direct public Blob URLs are visible to anyone with the URL; this is acceptable only because no auth/privacy layer exists yet.
- Group chat payloads can distinguish `chat_jid` and sender participant differently; parser changes should preserve both.
- Album payload shape may require additional extraction beyond the current single-media helper.
