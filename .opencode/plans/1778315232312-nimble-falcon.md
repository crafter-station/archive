# Scheduled Chat Log Agent Plan

## Decisions

- Use Trigger.dev scheduled task every 30 minutes: cron `*/30 * * * *` in timezone `America/Bogota`.
- Use OpenAI direct for the LLM, adding `OPENAI_API_KEY` and the current AI SDK/OpenAI provider after verifying local docs/source during implementation.
- Store all actual instants as timezone-aware UTC timestamps; keep `America/Bogota` local date/time metadata for display and event interpretation.
- `/[date]` uses hybrid grouping: logs, ships, and resources show on the Bogota day they were detected/shared; events show on the Bogota day they occur.
- Keep the UI server-rendered with link-based tabs and weekly navigation. No client state is required for the first version.

## Agent Window Semantics

Add `src/lib/log-windows.ts`.

- For scheduled timestamp `T`, current log window is `[T - 30m, T)`.
- Previous log window is `[T - 60m, T - 30m)`.
- Message context range is `[T - 60m, T)` because the prompt needs messages from the beginning of the previous log through the end of the current log.
- Store the current log row for `[T - 30m, T)`.
- Load the previous log row for `[T - 60m, T - 30m)` and copy its `finalResponse` into `logs.previousLogText`.
- Use half-open intervals to avoid counting boundary messages twice.
- Derive windows from the Trigger `payload.timestamp`, not `new Date()`, so late runs remain deterministic.

## Agent Implementation

Add these files:

- `src/trigger/chat-log-agent.ts`: exports the scheduled Trigger.dev task.
- `src/lib/log-agent.ts`: builds the system prompt, invokes the LLM, and returns final text.
- `src/lib/log-agent-queries.ts`: typed Drizzle reads/writes for logs, messages, memories, ships, resources, and events.
- `src/lib/log-agent-tools.ts`: AI tool definitions and Zod schemas that call the query helpers.

Task flow:

1. Compute current, previous, and context windows in `America/Bogota`.
2. Insert or reuse the `logs` row for the current window.
3. If the existing row is already `completed`, return early.
4. Mark the row `running` and attach `triggerRunId` when available.
5. Load previous log text.
6. Load messages from `[contextStartUtc, windowEndUtc)` with sender data.
7. Load active memories (`deletedAt is null`) and recent existing ships/resources/events for de-duplication.
8. Run the OpenAI-backed AI SDK agent with tools.
9. Let tools create/update/delete memories, create/update ships, create/update resources, and create/update events.
10. Store the final agent response in `logs.finalResponse` and mark the log `completed`.
11. On failure, mark `failed`, store the error, and rethrow so Trigger retries.

Trigger task shape:

- Use `schedules.task` from the same `@trigger.dev/sdk/v3` import style already used by the repo unless current Trigger docs require otherwise.
- Use cron `{ pattern: "*/30 * * * *", timezone: "America/Bogota" }`.
- Add `queue: { concurrencyLimit: 1 }` to prevent overlapping runs.
- Use `maxDuration: 300` initially unless LLM execution proves slower.
- Remove the sample `src/trigger/example.ts` or replace it with the real task export so only the useful schedule runs.

## Prompt And Tools

System prompt must include:

- Purpose: maintain a rolling community log and structured knowledge base from chat messages.
- Active memories injected as context.
- Previous log text injected as continuity context.
- Message context from the previous-window start through current-window end.
- Definition of a ship: a live internet artifact crafted by a community member, such as a GitHub repo, website, demo, Instagram/LinkedIn post, or similar.
- Definition of a resource: tutorial, book, documentation, product, or other link that can help others.
- Event time policy: infer dates/times in `America/Bogota` unless the message explicitly says another timezone/location; store UTC instant plus local metadata and uncertainty notes.
- De-duplication policy: prefer updating existing records over creating duplicates.
- Safety: only memories can be removed; ships/resources/events should be updated, not deleted.

Tools:

- `listMemories`
- `createMemory`
- `updateMemory`
- `removeMemory`
- `listShips`
- `createShip`
- `updateShip`
- `listResources`
- `createResource`
- `updateResource`
- `listEvents`
- `createEvent`
- `updateEvent`

Each create/update tool validates required fields with Zod and writes through Drizzle helpers. Tools do not expose raw SQL.
